中文标题：从 Terminal First 到 Browser First 的 Linux 工作流转变
English Title: From Terminal First to Browser First on Linux

本期视频回顾了作者重度使用 Arch Linux 与维护个人配置仓库近一年后的工作流变化。最初选择 Linux，是因为它的透明性、开源性、轻量性和高度可组合性；在早期阶段，作者投入大量时间研究窗口管理器、终端工具、键盘映射和多层配置，希望用一套统一的键盘操作逻辑控制大部分软件，并减少认知负担。这种 terminal first 的方式确实带来了轻量、快速、可脚本化和便于二次开发的优势，也符合“少就是多”的核心理念。

但随着使用深入，作者发现为了在不同终端应用、窗口管理器、底层按键映射和 GUI 软件之间维持统一操作，需要编写大量适配配置，甚至向上游提交补丁。不同软件对快捷键和交互模型的支持各不相同，导致为了追求简单统一，反而引入了更复杂的配置系统和维护成本。这个矛盾促使作者重新寻找平衡点，最终把主要工作流迁移到浏览器之上，形成 browser first 的方式。

视频重点说明了浏览器作为统一运行时的优势：Web 技术栈天然具有跨浏览器、跨系统的一致性，已经替用户完成了大量适配工作。作者从 Firefox 切换到 Chrome，并在浏览器之上使用键盘优先的插件，让链接点击、滚动、搜索、页面操作等都能通过统一快捷键完成。这样每个 tab 或页面都可以被看作一个应用，只需维护一份浏览器层面的键盘配置，就能控制大量 Web 应用，从而显著降低配置和记忆成本。

作者还举例说明了旧工作流如何被替代：过去会在终端中开启多个 OpenCode session，现在可以通过浏览器中的看板或 Web UI 统一管理多个对话；一些底层 CLI 或 SDK 能力也可以被包装成 Web 应用，用浏览器界面操控终端或本地服务的状态。音乐、视频、通信、在线文档、论文阅读和代码相关工作也都可以集中到浏览器工作空间中。这样一来，窗口管理器和终端工具的重要性下降，只要系统能稳定打开浏览器和少量尚未迁移的终端应用，就能满足大部分日常需求。

视频也讨论了 browser first 的局限。Web 应用通常比终端或原生应用更消耗资源，一个 tab 可能占用数百兆内存；对于图形计算、视频渲染等高性能专业场景，原生应用仍然更合适；部分 Web 服务依赖网络和平台厂商，不一定符合 local first 的理念。不过作者认为，在自己以写代码、读论文、沟通交流和轻量办公为主的场景中，这些缺点可以接受。相比极致性能和极简终端体验，当前更重要的是减少认知负担、降低维护成本、获得跨系统兼容性，并通过浏览器这一统一平台让操作更加快速、流畅和自然。
